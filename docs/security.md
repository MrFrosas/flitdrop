# Security model

## What is guaranteed

**End-to-end encryption between the phone (web page) and the computer.**

- Algorithm: XChaCha20-Poly1305 (AEAD), from the audited @noble/ciphers library, identical on the browser and the computer.
- Key: 32 random bytes generated by the computer at pairing, carried only in the **fragment** of the QR code URL (`/s/#id.key`). A URL fragment is never sent in HTTP requests, so the key never travels over the network, even in the clear locally.
- Every message carries a **contextual AAD** (`wd1|device|purpose|transfer|chunk#`): a captured ciphertext cannot be replayed on another route, for another device, or at another file position.
- JSON envelopes carry a timestamp (2-minute window) and a unique `jti` (anti-replay cache).
- A tampered, truncated or reordered chunk is rejected (Poly1305 tag + AAD). Tested.

**Locked admin surface.** The desktop interface and admin API are reachable only from the machine itself (loopback check) and only with a token (query once, then HttpOnly SameSite=Strict cookie). Constant-time comparisons. The real-time WebSocket requires the same token and loopback.

**Defensive disk writes.** File names are sanitized (path traversal, forbidden characters, Windows reserved names, bounded length), collisions de-duplicated, written to a `.tmp` then atomically renamed with an exclusive reservation. Max file size is enforced.

**Anti-abuse.** The chunk route rejects unknown devices before reading the body. Rate limits per IP. Bounded JSON bodies. Devices are revocable in one click (key destroyed).

## Accepted trade-offs (v1)

1. **The iOS Shortcut routes (`/api/shortcut/*`) use a bearer token, without application-layer encryption**, because the Shortcuts app cannot do crypto. On a hostile Wi-Fi, content sent by the Shortcut is readable by a local network observer. On your own Wi-Fi (the use case) the risk is low. The token is per-device and revocable. The web page is end-to-end encrypted everywhere.
2. **No TLS on the LAN**, on purpose: a self-signed certificate triggers blocking warnings on mobile. The application-layer AEAD covers confidentiality and integrity of the data; HTTP metadata (paths, sizes) stays visible on the LAN.
3. **Shared localhost.** Another program running under your session can read the admin token in `~/.flitdrop/config.json`. That is the standard desktop-app threat model (local malware has already won).

## Automated security tests

`npm test` covers: tampered chunk → 403, replayed envelope → 403, unknown device → 403, cross-context AAD → failure, stale timestamp → failure, invalid Shortcut token → 401, admin without token → 401, path traversal → neutralized, oversized file → 413, hard cap on the anti-replay cache, atomic file-name reservation.

## Adversarial attack review (July 2026)

A four-angle adversarial review (auth/crypto, filesystem, denial of service, web-client security), each finding independently counter-verified against the real code, found 3 false positives and **4 real defects, all fixed the same day**: a file-name race (now an atomic `O_CREAT|O_EXCL` reservation), body buffered before authentication (a device guard now runs before any body parser), an unbounded anti-replay cache (now a hard cap with amortized FIFO eviction), and a descriptor leak on an interrupted download (the disk stream is now destroyed as soon as the connection closes).
